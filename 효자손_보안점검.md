# 효자손 보안 점검 보고서

> K-AI Contents Award 지원서 '안전·보안' 항목 및 대회 유의사항(개인정보 무단 수집·보안 취약점 방치 금지) 대응 자료.
> 출시 전 SaaS 보안 점검 프롬프트(기술 리드·보안·결제·개인정보 관점, 13개 항목)를 실제 코드에 적용해 점검·수정한 기록.
> 점검일: 2026-07-04

---

## 1. 점검 방법
실제 배포될 코드(`api/*`, `public/js/*`)를 대상으로 13개 관점을 점검하고, 발견된 결함은 **실제 코드 경로로 재현 → 수정 → 재검증**했습니다. 추정이 아니라 실증 기반입니다.

## 2. 발견·조치 요약

| 위험도 | 항목 | 상태 |
|---|---|---|
| 🔴 높음 | 사진 속 악성 글자로 인한 XSS(스크립트 실행) | **수정·재검증 완료** |
| 🔴 높음 | 무인증 분석 API + 무력한 요청제한 → 비용 폭탄 | **공유 요청제한으로 수정** (+월 지출 한도 설정 권고) |

| 🟡 중간 | Cron 엔드포인트 무인증 노출 | **인증 필수로 수정** |
| 🟢 낮음 | 인증/권한/타 사용자 데이터/개인정보/백업/이메일 등 | 무저장·무수집 설계로 양호 |

## 3. 상세 (위험도 / 문제 / 사고 / 회피 / 수정 / 개선 프롬프트)

### 🔴 [높음] XSS — 사진 속 악성 글자로 화면 조작
- **문제**: AI가 사진에서 읽어낸 텍스트를 화면에 `innerHTML`로 그대로 삽입 → 사진에 `<img onerror=...>` 같은 명령 글자가 있으면 피해자 브라우저에서 스크립트 실행. 실제 코드 경로(기록 렌더)로 재현 확인.
- **생길 수 있는 사고**: 사기꾼이 만든 가짜 고지서 이미지를 어르신이 촬영 → 그 어르신의 기기 내 기록 탈취·화면 조작. 사기 예방 앱이 사기 도구로 악용되는 최악의 시나리오.
- **회피**: 신뢰할 수 없는 텍스트를 innerHTML에 직접 넣지 않기.
- **수정(적용)**: `esc()` 이스케이프 함수로 낱말풀이·핵심주의·할일·사실확인·문서종류 등 모델 유래 값 전부 처리. 재현 시도 시 스크립트 미실행·텍스트로만 표시 확인.
- **개선 프롬프트**: "이 프론트엔드에서 사용자/모델 유래 데이터가 innerHTML로 들어가는 모든 지점을 찾아 이스케이프하거나 textContent로 바꾸고, 누락이 없는지 검증해줘."

### 🔴 [높음] 비용 폭탄 — 무인증 API + 무력한 요청제한
- **문제**: `/api/analyze`가 로그인 없이 유료 비전 AI를 호출. 기존 요청제한은 서버 메모리 방식이라 서버리스에서 인스턴스마다 초기화되어 사실상 무력.
- **생길 수 있는 사고**: 스크립트로 반복 호출 → 월 API 요금 폭증.
- **회피**: 공유 저장소 기반 요청제한 + 모델 제공사 지출 한도.
- **수정(적용)**: Upstash KV 공유 요청제한(IP·시간당, 인스턴스 간 공유되어 유효)으로 교체, KV 장애 시 메모리 폴백. 한도는 복지관 공유 와이파이 다중 사용 고려해 시간당 100회(환경변수 `ANALYZE_RATE_LIMIT`로 조정).
- **남은 필수 조치(코드 밖)**: OpenAI·Gemini 콘솔에서 **월 지출 한도(spend cap)** 설정 — 요금 폭탄의 최종 방어선.
- **개선 프롬프트**: "무인증 공개 API 남용을 막도록 공유 저장소 기반 rate limit과 요청 크기 제한을 넣고, 모델 제공사 월 지출 한도 설정을 체크리스트로 정리해줘."

### 🟡 [중간] CSP(콘텐츠 보안 정책) 헤더 추가
- **문제/조치**: 응답 헤더에 `Content-Security-Policy`(`script-src 'self'` 등) 추가로 인라인/외부 스크립트 주입을 원천 차단(XSS 방어 심화). 앱이 외부 폰트·CDN 없는 자체 완결 구조라 부작용 없음. `/api/*`엔 `no-store` 캐시 헤더.

### 🟡 [중간] Cron 엔드포인트 인증
- **문제/조치**: `/api/refresh-blacklist`가 `CRON_SECRET` 미설정 시 공개돼 공공데이터 대량 fetch·KV 쓰기 유발 가능 → Vercel Cron 헤더 또는 시크릿 필수로 변경. 무인증 호출 401 확인.

### 🟢 [낮음] 나머지 관점
- **인증/권한(1·2)**: 회원가입·계정 없음 → 인증 취약점 자체가 없음.
- **타 사용자 데이터 접근(3)**: 서버 무저장(stateless), 기록은 각 기기 IndexedDB에만 → 서버 통한 교차 접근 불가.
- **개인정보 저장·삭제(5)**: 사진 즉시 폐기·서버 미저장, 기록은 텍스트만·개별/전체 삭제권, 처리방침 구현.
- **관리자 페이지(6)**: 없음.
- **API 키/환경변수(7)**: 키는 서버 환경변수에만, 클라이언트 미전송, `.env`/`.env.vercel` gitignore, 오류 응답 일반화. 정상.
- **에러/장애(9)**: 오류 시 일반 문구, 이미지 미로깅, 폴백 모델. (모니터링은 추후.)
- **배포/백업(10)**: DB 없음(무상태), KV 캐시는 Cron 재생성 가능 → 백업 불필요.
- **이메일(11)**: 발송 기능 없음.
- **약관/개인정보/환불(12)**: 이용약관·개인정보 처리방침 구현. 무료라 환불 N/A.
- **추가**: QR은 디코딩만 하고 접속 안 함(SSRF 없음), 사용자 지정 URL fetch 없음.

## 4. 론칭 전 테스트 시나리오 (13번)
1. 모델 제공사 월 지출 한도 설정 후 초과 동작 확인 (최우선)
2. 조작 이미지(HTML/스크립트 글자) 업로드 → 스크립트 미실행 (XSS 회귀)
3. 같은 IP 시간당 한도 초과 → 429 안내 (공유 rate limit)
4. `/api/refresh-blacklist` 무인증 호출 → 401
5. 실기기(iOS/Android) 카메라·저장·카톡 공유 (HTTPS)
6. 흐린/글자 없는 사진 → "다시 찍어주세요"
7. 오프라인·느린 네트워크 → 오류 문구·TTS 폴백

## 5. 지원서에 그대로 쓸 수 있는 '안전' 서술 (초안)

> **효자손은 '사기 예방' 앱인 만큼, 스스로 안전하도록 설계했습니다.**
> 출시 전 인증·권한·데이터 접근·결제·개인정보·비용·배포 등 13개 관점의 보안 점검을 실제 코드에 적용해 수행했습니다.
> ① **개인정보 비수집**: 회원가입이 없고 서버는 아무것도 저장하지 않으며(stateless), 촬영한 사진은 분석 후 즉시 폐기됩니다. 기록은 이용자 기기에만 남고 언제든 삭제할 수 있습니다.
> ② **취약점 실증 점검**: 사진 속 악성 글자로 인한 스크립트 실행(XSS)을 실제로 재현·차단(이스케이프 + CSP)했고, 무인증 API 남용에 의한 비용 폭탄을 공유 요청제한과 지출 한도로 방어했습니다.
> ③ **키·엔드포인트 보호**: API 키는 서버에만 두고 클라이언트에 노출하지 않으며, 관리용 자동화 엔드포인트는 인증을 요구합니다.
> 이러한 설계로 대회 유의사항(개인정보 무단 수집·보안 취약점 방치 금지)을 정면으로 충족합니다.
